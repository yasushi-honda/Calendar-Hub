## 副次的発見

### 1. Firestore transaction の optimistic locking 不全 (AC-E2E-2) → 本 PR で修正済

`apps/api/src/routes/public-booking.ts:269-300` の `runTransaction` 内で `tx.get(...)` ではなく `db.collection('bookings').where(...).get()` を使用しており、Firestore の serializable isolation が効かない不具合を E2E が検出。

- 本番影響: 公開リンクへの並列予約で実発生し得るデータ整合性破壊 (実害基準 #1)
- 対処: 本 PR 内で `tx.get(overlapQuery)` に置換修正。AC-E2E-2 spec を有効化して PASS 確認済。

### 1.5 E2E mock 機構の本番事故防止 guard (`assertE2EMockSafe`) → 本 PR で追加

`E2E_MAIL_MOCK=1` / `E2E_CALENDAR_MOCK=1` が誤って本番 Cloud Run に設定された場合、Gmail / Google Calendar への実送信が黙って bypass され Firestore `_e2eMail` collection に書かれる silent disaster になりうる。`apps/api/src/lib/e2e-guard.ts` に `assertE2EMockSafe(flagName)` を新設し、3 箇所 (email.ts / adapter-factory.ts / public-booking.ts) と `app.ts` の ratelimit 分岐で `NODE_ENV === 'production'` 検知時に fail-fast throw する形にした。E2E モードでは公開予約 API のレート制限も自動 disable される。

### 2. 過去 slot を `/slots` API が返している (低影響)

実装側で `slots` 配列に現在時刻より前の slot が含まれるケースを確認。UI 側で「過去」と判定されるため UX 上は問題ないが、レスポンスの cleanliness としては修正余地あり。本 PR スコープ外。

### 3. `pnpm.overrides` で minimatch v3 が壊れていた (root cause: brace-expansion v5 化)

`brace-expansion@>=4.0.0 <5.0.5: >=5.0.5` の override で minimatch@3.1.5 の `braceExpand()` が `expand is not a function` で fail。`minimatch@<5.0.0: >=5.0.0` の override を追加して v5 に強制 upgrade することで解消。security override の連鎖で発生したリグレッションで、E2E 導入とは無関係だが本 PR で発見・修正した。

### 4. `next lint` 既存テスト spec を vitest が誤検出

`vitest.config.ts` を新規作成して `apps/web/e2e/**` と `dist/**` を exclude (dist 重複カウントによる 305→213 件 normalization も同時実施)。

## 既知の制約

- ポーリング検証 spec が 2.2 分かかる (60s 待機 × 2 が必要)。CI 全体時間 +3 分程度の見込み。
- `firebase emulators:exec` 経由でしか e2e 環境変数を渡せないため、`pnpm e2e` が唯一の実行経路。
- リンク作成 UI (認証フロー込み) の E2E はスコープ外。実装するには Auth Emulator の test user seed + ログイン状態シミュレーションが必要。
